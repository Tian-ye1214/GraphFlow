# GraphFlow 压测 / 冲突测试报告

**日期**：2026-06-20
**范围**：数据库、链路并发、超大文件三大维度的压力测试与冲突测试；附链路导入导出 `.gfpkg` 功能活体复验。
**结论先行**：三维度全部通过，无新缺陷、无 HTTP 500、无数据丢失、无跨租户泄漏；瓶颈点是 SQLite 写串行（WAL 单写者）与逐行 LLM 调用（受模型 RTT 支配，节点内并发线性提速）。

---

## 环境与方法

- **被测**：运行中的真实后端 `http://127.0.0.1:8000`（合并 master@313cbc4 的链路导入导出功能）+ 真实 SQLite(WAL) DB + 真实 DeepSeek（`deepseek-v4-pro`）。
- **测法**：Python + httpx 直连（`trust_env=False` 避开系统代理）；并发用线程池（每线程独立客户端/会话）精确编排；计时用 `time.perf_counter()`。
- **安全**：全程用 `smoke_` 前缀资源，建即删；绝不触碰 zrs 真实数据；测后只读核验基线精确还原。
- **全套总耗时**：91.3 s。

> 说明：上传行数据列为 4 列（`q`/`idx`/`c0`/`c1`）；LLM 测试关思考、`max_tokens` 给足以避免推理模型空内容；超大文件的“跑链路”用 `auto_process`（纯计算、无 LLM）以在 10 万行规模上测吞吐而不产生海量 token 成本——逐行 LLM 并发另在“链路并发”小规模单测。

## 一、数据库

### 1.1 批量写吞吐（单连接串行上传）

| 行数（×4 列） | 上传耗时 | 吞吐 |
|---:|---:|---:|
| 1,000 | 25 ms | ~39,400 行/s |
| 5,000 | 206 ms | ~24,200 行/s |
| 20,000 | 393 ms | ~50,900 行/s |
| 50,000 | 1,009 ms | ~49,600 行/s |

稳态约 **5 万行/s**（含 HTTP + 解析 + `executemany` 批量插入 + commit）。5,000 行那档单行偏慢系冷启动/抖动，规模上去后线性。

### 1.2 WAL 并发写争用（冲突测试）

5 个独立用户**同时**各上传 5,000 行：**并发 684 ms vs 串行 551 ms（并发/串行 = 1.24）**。

- 解读：SQLite WAL 是**单写者**模型，写入本质串行；并发不仅没提速，反因写锁排队 + 协调略增 24% 开销。**但全程无死锁、无 `database is locked` 失败、无丢行**——这是 SQLite 写并发的正确预期行为。
- 含义：写密集场景的扩展瓶颈在 DB 写串行，不在应用层；如需更高写并发应换 Postgres，当前规模 SQLite 足够。

### 1.3 并发下租户隔离（冲突测试，红线）

5 用户并发上传后，逐个核验各自 `/api/datasets` 列表：**每个用户只看到自己的 2 个数据集，越界泄漏用户数 = 0**。即便并发写同名数据集（`smoke_db_conc`/`smoke_db_seq`），归属仍按 `user_id` 严格隔离，无串号。

## 二、链路并发

### 2.1 节点内逐行并发（真实 DeepSeek）

同一 8 行 LLM 翻译链路，仅改 `concurrency`：

| concurrency | 耗时 | 状态 |
|---:|---:|:--|
| 1 | 64.2 s | completed |
| 8 | 11.4 s | **completed，5.64× 加速** |

逐行隔离 + 节点内信号量并发生效；耗时由模型 RTT 支配，并发近线性提速（8 路未满 8× 系受用户级 LLM 信号量与模型侧限流影响）。

### 2.2 多 run 同时（并发编排）

同一 `auto_process` 链路**同时发起 5 个 run**：全部 `completed`，总耗时 1.3 s。多运行并行调度无相互阻塞、无错乱。

> 架构事实：节点级执行串行，并发只发生在“节点内逐行”；多个 run 之间各自独立并行。本测两点都验到。

## 三、超大文件

数据集 **100,000 行 × 4 列 = 7.3 MB** JSONL：

| 阶段 | 耗时 | 备注 |
|---|---:|---|
| 上传（解析 + 入库 10 万行） | 2.14 s | ~46,600 行/s |
| 读页 `GET /rows?page_size=100` | 16 ms | total=100,000，分页不被总量拖慢 |
| **导出 `.gfpkg`（流式打包全量行）** | 2.09 s | 包 1.0 MB（zip 把 7.3 MB 压到 1.0 MB） |
| **导入 `.gfpkg`（流式建集 + 批量插入）** | 2.04 s | 重建 10 万行数据集，单事务 |
| 跑链路 `auto_process(dedup)` 全量 | 2.9 s | barrier 单 RunRow 装全量、completed |

导出/导入均走**流式**（导出 `session.stream` 逐行写 zip 条目；导入 `TextIOWrapper` 逐行 + 1000/批 `executemany`），内存有界；10 万行端到端（上传→导出→导入→跑）合计 ~9 s。读压（大数据集分页读回）16 ms，无退化。

## 附：链路导入导出 `.gfpkg` 功能活体复验（真实 DeepSeek）

15 项端到端检查 **14 通过**：上传数据集、真实 DeepSeek 跑通（你好→Hello、苹果→apple、水→water，3.0 s）、导出 200/zip、模型导出不含 api_key、数据集行保真、导入 zrs 复用模型+数据集、导入后复用带 key 模型再跑通、http 头/url/body 脱敏、**包字节内零明文密钥**、导入报告点名待回填密钥位、跨账号导入建模型+新数据集、租户隔离（smoke 用户链路 zrs 不可见 / zrs 链路 smoke 不可达）、zrs 基线精确还原。

唯一 1 项不通过为**陈旧服务器假象**：跨账号导入新建的“空 key 模型”在重启前的旧码里用 `encrypt("")` 存储（密文非空）→ `api_key_set` 误报 `True`。已修（改存 `""`，对齐 `create_model` 约定，`api_key_set=bool(enc)=False`、运行时 `decrypt` 返回占位优雅失败），35 项包单测验证；**重启后端后该项即转通过**。

## 结论与瓶颈

| 维度 | 结论 | 瓶颈 |
|---|---|---|
| 数据库 | 写吞吐 ~5 万行/s，并发零死锁/零丢失/零越界 | SQLite WAL 单写者（写串行，并发不提速） |
| 链路并发 | 逐行并发 5.64× 提速、多 run 并行无阻塞 | 逐行 LLM 受模型 RTT/限流支配 |
| 超大文件 | 10 万行上传/导出/导入/跑链路各 ~2 s，流式有界 | 无明显瓶颈（当前规模） |

无需代码改动（除活体已修的 1 处 keyless 模型 api_key_set 误报）。如未来写并发或单数据集行数再上一个数量级（百万级），建议评估 Postgres 与导出/导入分片。
